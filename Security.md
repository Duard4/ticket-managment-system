# Security Policy — Ticket Management System (TMS)

## Контроль доступу

TMS реалізує рольову модель доступу (RBAC). Кожен користувач має чітко визначену роль, яка обмежує доступні функції та дошки.

### Матриця прав доступу

| Функція                    | Гість | Користувач | Оператор | Адміністратор |
| -------------------------- | ----- | ---------- | -------- | ------------- |
| Перегляд External Board    | —     | +          | +        | +             |
| Перегляд Internal Board    | —     | —          | +        | +             |
| Створення заявки           | —     | +          | +        | +             |
| Редагування власної заявки | —     | ±          | +        | +             |
| Редагування чужих заявок   | —     | —          | +        | +             |
| Зміна статусу заявки       | —     | —          | +        | +             |
| Видалення записів          | —     | —          | —        | +             |
| Керування користувачами    | —     | —          | —        | +             |
| Перегляд журналу подій     | —     | —          | ±        | +             |
| Налаштування SLA           | —     | —          | —        | +             |

`+` — дозволено, `—` — заборонено, `±` — частково дозволено

### Реалізація

- Авторизація через **JWT-токени** (термін дії: 24 год).
- Паролі зберігаються у хешованому вигляді — **bcrypt** (salt rounds: 10).
- Guards реалізовані на рівні NestJS-контролерів (`src/common/guards`).
- Секретні ключі зберігаються виключно у `.env` (виключено з репозиторію через `.gitignore`).

## Відомі вразливості

### BUG-001 — Bypass guard при прямому URL-доступі до Internal Board

| Поле                    | Опис                                                                                                                                          |
| ----------------------- | --------------------------------------------------------------------------------------------------------------------------------------------- |
| **ID**                  | BUG-001                                                                                                                                       |
| **Серйозність**         | Critical                                                                                                                                      |
| **Статус**              | Open (в роботі)                                                                                                                               |
| **Опис**                | Guard-механізм не перевіряє роль користувача при прямому переході за URL `/internal-dev-board`. External-користувач бачить внутрішній Kanban. |
| **Очікувана поведінка** | HTTP 403 Forbidden + редірект на External Board                                                                                               |
| **Фактична поведінка**  | HTTP 200 OK, сторінка завантажується                                                                                                          |
| **Пов'язаний issue**    | [#1 — Fix BUG-001: guard bypass](../../issues/1)                                                                                              |

## Що не можна зберігати в репозиторії

- Файл `.env` з реальними паролями та секретами
- JWT-секрети, API-ключі, токени сервісів
- Дампи бази даних з реальними даними користувачів
- Приватні SSL-сертифікати

Для перевірки використовується [git-secrets](https://github.com/awslabs/git-secrets) або GitHub Secret Scanning.

## Журналювання подій безпеки

Система фіксує наступні події, пов'язані з безпекою:

| Подія                                | Рівень    | Що фіксується                      |
| ------------------------------------ | --------- | ---------------------------------- |
| Успішний вхід                        | Середній  | Час, логін, IP                     |
| Невдала спроба входу                 | Високий   | Час, логін, IP, кількість спроб    |
| Доступ до забороненого ресурсу (403) | Критичний | URL, метод, IP, час                |
| Зміна ролі користувача               | Високий   | ID, стара/нова роль, адміністратор |
| Скидання пароля                      | Середній  | ID користувача, час, IP            |

> Паролі, токени та персональні дані у відкритому вигляді в логи **не записуються**.

Журнали безпеки зберігаються **не менше 365 днів** та доступні лише Адміністратору.

## Повідомлення про вразливості

Якщо ви виявили вразливість у системі:

1. **Не створюйте публічний issue** з описом вразливості.
2. Повідомте через GitHub: відкрийте **[Security Advisory](../../security/advisories/new)**.
3. Або напишіть безпосередньо розробнику через GitHub-профіль.

Будь ласка, включіть у повідомлення: опис вразливості, кроки відтворення, очікуваний та фактичний результат, середовище.
