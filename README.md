# Ticket Management System (TMS)

## 📌 Опис проєкту

Ticket Management System — це інформаційна система, призначена для автоматизації процесів реєстрації, обробки та відстеження звернень щодо супроводу розгорнутого програмного продукту. Система підтримує повний життєвий цикл заявки: від моменту створення до остаточного вирішення та закриття.

## 🚀 Ключові функції

- **Гібридна обробка звернень:**
  - `External Support Board` — призначена для клієнтів та зовнішніх користувачів (запити на технічну підтримку).
  - `Internal Development Board` — робочий простір для команди розробки (баг-репорти, feature-запити від QA, Dev, BA).
- **Типізація заявок:** автоматична або ручна класифікація за категоріями: `Bug Report`, `Feature Request`, `Technical Support`.
- **Інтелектуальна маршрутизація\*:** автоматичне призначення заявок на відповідний відділ або дошку залежно від ролі автора та категорії запиту.
- **Контроль статусів:** відстеження станів заявки (`Нова` → `В обробці` → `Очікує відповіді` → `Вирішена`) з веденням повного журналу дій.
- **SLA-ескалація\*:** автоматичне сповіщення адміністратора при перевищенні терміну обробки.
- **Email-сповіщення\*:** підтвердження реєстрації заявки та оновлення статусів.

> \* — функціонал у розробці.

## 📂 Структура проєкту

```
├── backend/                  # Серверна логіка (API, маршрутизація, БД)
│   ├── src/modules/tickets   # Обробка об'єктів "Заявка"
│   ├── src/modules/boards    # Логіка Internal vs External дошок
│   └── src/common/guards     # Розмежування прав доступу (External/Internal)

    ├── src/features/external-support  # Інтерфейс для клієнтів
    ├── src/features/internal-dev      # Kanban-борд для розробників
    └── src/types                      # Спільні інтерфейси (Bug, Feature, Support)
```

## 👥 Ролі користувачів

| Роль | Опис | Доступ |
|------|------|--------|
| **Кінцевий користувач** | Зовнішній клієнт, що подає заявки | External Support Board |
| **Оператор** | Приймає та обробляє заявки | External + Internal Board |
| **Адміністратор** | Налаштовує систему, ролі, SLA | Повний доступ |
| **Керівник підрозділу** | Переглядає аналітику та звіти | Звіти + обидві дошки |

Детальна матриця прав доступу описана у [SECURITY.md](./SECURITY.md).

## ⚙️ Бізнес-правила

1. **Пріоритетність:** заявки з позначкою `Bug Report` на внутрішній дошці мають вищий пріоритет для QA та Dev.
2. **Безпека:** зовнішні користувачі не мають доступу до `Internal Development Board`.
3. **Валідація:** закриття заявки неможливе без фінального коментаря виконавця або підтвердження від ініціатора.
4. **Ескалація:** автоматичне сповіщення адміністратора у разі перевищення терміну обробки заявки.

## 🚀 Швидкий старт

### Вимоги

- Node.js v20+
- npm v9+
- PostgreSQL 16
- Git

### Встановлення

```bash
# 1. Клонувати репозиторій
git clone https://github.com/parkhomenko/tms.git
cd tms

# 2. Встановити залежності backend
cd backend && npm install

# 3. Встановити залежності frontend
cd ../frontend && npm install
```

### Налаштування середовища

Скопіювати `.env.example` → `.env` і заповнити змінні:

```env
DB_HOST=localhost
DB_PORT=5432
DB_NAME=tms_db
DB_USER=postgres
DB_PASS=yourpassword
JWT_SECRET=your_jwt_secret_min_32_chars
PORT=3000
FRONTEND_URL=http://localhost:5173
```

### Запуск

```bash
# Створити базу даних
createdb tms_db

# Виконати міграції
cd backend && npm run migration:run

# Запустити backend (порт 3000)
npm run start:dev

# В окремому терміналі — запустити frontend (порт 5173)
cd ../frontend && npm run dev
```
